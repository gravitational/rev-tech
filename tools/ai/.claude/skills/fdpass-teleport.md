# fdpass-teleport -- Teleport SSH File Descriptor Passing Helper

fdpass-teleport is a helper binary that integrates tbot's ssh-multiplexer service with OpenSSH. It delivers higher performance and reduced resource consumption by using Unix domain socket file descriptor passing instead of piping data through a ProxyCommand subprocess. Users should not need to invoke it manually -- it is automatically wired into the `ssh_config` generated by tbot's `ssh-multiplexer` service.

**Binary:** `fdpass-teleport` (Teleport v18.2.1+)
**Documentation:** <https://goteleport.com/docs/reference/cli/fdpass-teleport/>

---

## Quick Reference

### Syntax

```bash
fdpass-teleport <path to mux socket> <connection target>
```

### Arguments

| Argument | Description |
|----------|-------------|
| `<path to mux socket>` | Path to the Unix domain socket created by tbot's ssh-multiplexer service (e.g., `v1.sock` in the destination directory) |
| `<connection target>`  | The target address for the SSH connection, forwarded to the ssh-multiplexer service |

There are no subcommands, flags, or environment variables. This is a minimal single-purpose binary. All invocations without exactly two arguments (including `--help`, `--version`) print the usage banner and exit with code 1.

---

## Architecture

```text
OpenSSH --> fdpass-teleport --> v1.sock (ssh-multiplexer) --> Teleport Proxy --> SSH nodes
```

Instead of proxying SSH data through stdin/stdout like a traditional `ProxyCommand`, `fdpass-teleport` passes the file descriptor for the connection directly to the ssh-multiplexer via the Unix socket. This avoids the overhead of a long-lived subprocess per connection and is significantly more efficient for workloads with many concurrent SSH sessions (e.g., Ansible).

---

## How It Works with tbot ssh-multiplexer

The `ssh-multiplexer` is a long-running tbot service that:

- Opens a local Unix socket (`v1.sock`) implementing SSH multiplexing
- Multiplexes many SSH connections through fewer upstream connections to the Teleport Proxy Service
- Opens an SSH agent socket (`agent.sock`) so private keys are never written to disk
- **Cannot** operate in one-shot mode -- must run continuously on the same host as the SSH client

### tbot ssh-multiplexer Configuration

```yaml
services:
  - type: ssh-multiplexer
    destination:
      type: directory
      path: /opt/machine-id/ssh-mux
    # enable_resumption: true          # SSH session resumption (default: true)
    # proxy_command: ["fdpass-teleport"]  # custom ProxyCommand binary path
    # proxy_templates_path: /etc/tbot/proxy-templates.yaml
    # relay_server: relay.example.com:443
```

### ssh-multiplexer Configuration Options

| Option | Description |
|--------|-------------|
| `destination` | Directory for output artifacts -- `ssh_config`, `known_hosts`, `v1.sock`, `agent.sock` (required) |
| `enable_resumption` | Tolerance for network interruptions (default: true) |
| `proxy_command` | Custom ProxyCommand binary path array (default: `["fdpass-teleport"]`) |
| `proxy_templates_path` | Path to proxy template configuration file |
| `relay_server` | Alternative routing address |
| `name` | Custom service identifier |

### ssh-multiplexer Output Artifacts

| File | Purpose |
|------|---------|
| `ssh_config` | OpenSSH configuration referencing the multiplexer and fdpass-teleport |
| `known_hosts` | Server identity validation |
| `v1.sock` | Unix socket for the multiplexer listener |
| `agent.sock` | Unix socket for SSH agent functionality |

### Starting ssh-multiplexer via CLI

```bash
# With config file
tbot start -c /etc/tbot.yaml

# Directly via CLI flags
tbot start ssh-multiplexer \
  --proxy-server=example.teleport.sh:443 \
  --join-method=kubernetes \
  --token=ssh-bot \
  --destination=file:///opt/machine-id/ssh-mux \
  --enable-resumption \
  --proxy-command=fdpass-teleport \
  --proxy-templates-path=/etc/tbot/proxy-templates.yaml
```

## Usage Examples

### Basic SSH via tbot-generated ssh_config

```bash
# tbot generates the ssh_config with fdpass-teleport wired in
ssh -F /opt/machine-id/ssh-mux/ssh_config user@node.teleport-cluster
```

The user never invokes `fdpass-teleport` directly -- the generated `ssh_config` handles all the wiring.

### With Ansible

The ssh-multiplexer + fdpass-teleport combination is designed for workloads that create a large number of SSH connections. Configure Ansible to use the tbot-generated ssh_config:

```ini
# ansible.cfg
[ssh_connection]
ssh_args = -F /opt/machine-id/ssh-mux/ssh_config
```

### Specifying a Custom fdpass-teleport Path

If `fdpass-teleport` is not on `$PATH`, specify the full path in the tbot configuration:

```yaml
services:
  - type: ssh-multiplexer
    destination:
      type: directory
      path: /opt/machine-id/ssh-mux
    proxy_command:
      - /usr/local/bin/fdpass-teleport
```

### Installing fdpass-teleport

```bash
# Included in Teleport tar download install script since v18.2.1

# Or copy from tbot binary
tbot copy-binaries --include-fdpass /usr/local/bin
```

## Troubleshooting

### Connection Fails Immediately

- **ssh-multiplexer not running:** Verify tbot is running with the ssh-multiplexer service. The `v1.sock` file must exist in the destination directory.
- **Socket path mismatch:** Ensure the `ssh_config` references the correct path to `v1.sock`. Check `cat /opt/machine-id/ssh-mux/ssh_config` and verify the ProxyCommand line.
- **fdpass-teleport not found:** Verify the binary is installed at the path specified in `proxy_command`. Run `which fdpass-teleport` or check the full path.

### Permission Denied on Socket

- Ensure the SSH client user has read/write access to `v1.sock` and `agent.sock`.
- Run `tbot init` to set up proper ownership and ACLs on the destination directory.

### OpenSSH Does Not Support ProxyUseFdpass

- Verify OpenSSH version: `ssh -V` (requires 9.4+).
- If using an older OpenSSH, the ssh-multiplexer falls back to a standard ProxyCommand behavior without fd passing. Consider upgrading OpenSSH.

### SSH Client Library Compatibility

- Not all SSH client libraries support fd passing. If using a programmatic SSH client, verify it supports `ProxyUseFdpass` or `ProxyCommand` with fd passing before relying on this integration.

### Enable Debug Logging

```bash
# Debug tbot (shows ssh-multiplexer service activity)
tbot start -c /etc/tbot.yaml --debug
```
